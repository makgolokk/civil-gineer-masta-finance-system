# CGM Export Backend Production Plan

The Vercel app is a static frontend. Professional PDF and Excel generation is handled by the FastAPI backend in `backend/app`, which should run as a separate Python service.

## Recommended Deployment

1. Deploy the Python service with Docker using `backend/Dockerfile`.
2. Set environment variables:
   - `PORT=8765` or the platform-provided port
   - `CGM_EXPORT_ENVIRONMENT=production`
   - `CGM_EXPORT_ALLOWED_ORIGINS=https://your-vercel-app.vercel.app`
   - `CGM_EXPORT_FRONTEND_ORIGIN=https://your-vercel-app.vercel.app`
   - `CGM_EXPORT_ADDITIONAL_ALLOWED_ORIGINS=` if extra office origins are required
3. Confirm the service responds at:
   - `/health`
   - `/exports/quotation`
   - `/exports/invoice`
   - `/exports/receipt`
   - `/exports/client-statement`
   - `/exports/excel`
4. Add the hosted service URL to Vercel:
   - `VITE_EXPORT_API_BASE_URL=https://your-export-service.example.com`
5. Redeploy the Vercel frontend.

## Good Hosting Options

- Render Web Service using the Dockerfile
- Railway Docker service
- Fly.io Docker app
- A small VPS with Docker and HTTPS reverse proxy

## Why Separate Hosting

The export backend uses ReportLab and OpenPyXL. Keeping it separate from the static Vercel frontend gives better control over Python dependencies, memory, file generation, and future server-side permissions.

## Client-Ready Export Rule

Known client documents must be generated by the professional backend. If the backend is unavailable, the frontend shows a clear error instead of downloading raw field/value quotation, invoice, receipt, or client statement PDFs.

## Next Security Step

Before real multi-user use, protect export endpoints so staff can only export records they are allowed to view.
